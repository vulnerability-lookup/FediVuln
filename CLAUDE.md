# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

FediVuln is a Mastodon (Fediverse) client that bridges the Fediverse and a [Vulnerability-Lookup](https://github.com/vulnerability-lookup/vulnerability-lookup) instance, in both directions:

- **Inbound (Fediverse → Vulnerability-Lookup)**: `FediVuln-Stream` listens to Mastodon streams, scans statuses for vulnerability identifiers, and creates *sightings* in Vulnerability-Lookup.
- **Outbound (Vulnerability-Lookup → Fediverse)**: `FediVuln-Publish` subscribes to a Vulnerability-Lookup event stream (HTTP SSE or Valkey Pub/Sub) and posts formatted statuses on Mastodon.

The package is managed with Poetry and ships four console entry points (see `pyproject.toml` `[project.scripts]`): `FediVuln-Register`, `FediVuln-Stream`, `FediVuln-Search`, `FediVuln-Publish`.

## Commands

Dev environment uses Poetry with an optional `dev` group.

```bash
poetry install --with dev          # install with dev tools (mypy, flake8, pre-commit)
poetry run pre-commit run --all-files
poetry run mypy fedivuln           # type check (configured in pyproject.toml, target Python 3.13)
poetry run flake8 --max-line-length=120 fedivuln
poetry build                       # build sdist + wheel
```

There is **no test suite** in this repo — don't invent one or claim coverage. Manual smoke testing is done via the CLI entry points against a real Mastodon instance and a real Vulnerability-Lookup instance.

Pre-commit chain: `pyupgrade --py37-plus`, `isort` (black profile), `black`, `flake8` (with `flake8-bugbear`, `flake8-implicit-str-concat`, max line 120), `pip-audit`. Match this — do not introduce a different formatter.

## Configuration model

Configuration is **a Python file** loaded dynamically by `fedivuln/config.py` from the path in `$FEDIVULN_CONFIG` (defaults to `fedivuln/conf_sample.py`). `config.py` imports specific attributes from that module — when adding a new config knob, you must update both the loader (`config.py`) and the sample (`conf_sample.py`), and decide whether the attribute is required (raises if missing) or optional (caught `AttributeError`, default applied). The current required/optional split:

- **Required**: `api_base_url`, `scopes`, `app_name`, `mastodon_clientcred`, `mastodon_usercred`, `templates`, `status_visibility`, `vulnerability_patterns`, `vulnerability_lookup_base_url`, `vulnerability_auth_token`.
- **Optional**: `mastodon_clientcred_push` / `mastodon_usercred_push` (a *second* Mastodon account used by `publish.py` if set, otherwise it falls back to the primary one), `valkey_host` / `valkey_port` / `expiration_period` (gate `heartbeat_enabled`).

Mastodon credentials are stored as files (`mastodon_clientcred.secret`, `mastodon_usercred.secret`) generated by `FediVuln-Register`, which performs the OAuth dance once.

## Architecture notes

- **Vulnerability ID detection** lives entirely in `conf_sample.py` (`vulnerability_patterns`), not in code. It is a single compiled regex with multiple named alternatives (CVE, GCVE, GHSA, PYSEC, GSD, CERT-Bund, Cisco, RHSA, MSRC, CERT-FR). To support a new identifier scheme, extend that regex; `stream.py` flattens all capture groups generically.
- **Status templating** is also config-driven: `templates[topic]` contains placeholders like `<VULNID>`, `<VENDOR>`, `<DESCRIPTION>`, `<VLAI-SCORE>`. `publish.create_status_content` is a `match topic:` dispatch that fills these. Mastodon's 500-char limit (`MAX_STATUS_LENGTH`) is enforced by computing remaining space *after* substituting all other placeholders, then truncating the description with `utils.truncate`.
- **VLAI severity scoring** is fetched from `https://vulnerability.circl.lu/api/vlai/severity-classification` per published vulnerability (see `utils.classify_vulnerability_severity`, async via `aiohttp`). `publish.py` runs it via `asyncio.run` inside an otherwise sync flow — keep that boundary; don't make the SSE loop async without restructuring.
- **Two ingestion paths** in `publish.py`: HTTP SSE (`listen_to_http_event_stream`, default) reads `text/event-stream` from `{vulnerability_lookup_base_url}/pubsub/subscribe/{topic}` with the `X-API-KEY` header; Valkey (`--valkey`, `listen_to_valkey_stream`) is a generator subscribing to a Pub/Sub channel named after the topic.
- **Observability** is a tiny Valkey-based heartbeat/log layer (`fedivuln/monitoring.py`). `heartbeat()` sets a TTL key per process; `log()` rpushes structured strings to a 24h list. `heartbeat_enabled` only flips on if Valkey config is present — code paths must tolerate it being off.
- **Vulnerability-Lookup client** (`PyVulnerabilityLookup`) is constructed *per push* in `stream.push_sighting_to_vulnerability_lookup` rather than reused — preserve that unless you have a reason to introduce a long-lived client.

## Docker

`Dockerfile` builds an image whose entrypoint is hardcoded to `FediVuln-Stream --public --push-sighting`. `docker-compose.yml` mounts `./fedivuln/conf.py` to `/etc/conf.py` and sets `FEDIVULN_CONFIG=/etc/conf.py`. The image also `COPY`s `mastodon_clientcred.secret` / `mastodon_usercred.secret` from the build context — these must be generated locally first (`FediVuln-Register`).
